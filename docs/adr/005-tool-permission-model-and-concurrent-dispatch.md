# ADR-005：工具权限模型与并发调度

## 状态

已接受

## 背景

`AgentLoop` 在一次 LLM 响应中可能收到多个 `ToolUseContent`。如果简单地把所有工具调用都交给 `asyncio.gather(...)` 并发执行，短期内代码最少，但很快会遇到三类问题：

1. 并发写操作会产生竞争  
   两个工具同时写文件、修改外部状态或操作同一资源时，很容易造成顺序不确定和结果互相覆盖。

2. 不同工具的风险等级不同  
   `read_file`、`http_get` 与 `write_file`、`shell_exec` 的安全语义并不相同。如果 runtime 不显式建模权限，所谓“工具调用”就只是一个缺乏安全边界的任意执行入口。

3. 模型可能一次吐出大量工具调用  
   如果没有并发上限，LLM 一旦进入失控或低质量推理状态，宿主机资源会被瞬间打满，runtime 无法自我保护。

因此，项目需要一个既能表达权限边界、又能控制并发策略的工具执行模型。

## 决策

我们决定将“权限模型”和“调度策略”作为 runtime 的一等概念处理，并采取以下约束：

1. 定义三档工具权限：
   - `read_only`
   - `write`
   - `dangerous`
2. `AgentLoop` 在启动时显式声明 `allowed_permissions` 白名单。
3. Tool schema 暴露给模型之前，先按权限白名单过滤，不允许的工具不会出现在 schema 中。
4. 调度策略按权限分层：
   - `read_only` 工具可并发执行
   - `write` 与 `dangerous` 工具按出现顺序串行执行
5. 通过 `max_concurrent_tools` 和 semaphore 实现全局并发上限。
6. 调度实现从 `AgentLoop` 中拆出，独立为 `ToolDispatcher`。

## 理由

### 1. 权限是 runtime 的契约，不是业务层约定

如果工具是否危险只存在于注释或调用方自觉中，最终一定会退化成“谁都能调任何东西”。把权限作为 `Tool` 契约的一部分，才能让 runtime 在 schema 暴露、执行审批和并发策略上做统一控制。

### 2. 对只读工具保留并发，性能收益最大

多数 agent 工具调用都偏 I/O 型。对于 `read_only` 工具，并发通常是安全且有收益的，例如同时发起多个 `http_get` 或多个只读文件查询。这部分并发可以显著降低 step 内等待时间。

### 3. 对写操作串行化，是更稳的默认值

`write` 和 `dangerous` 工具如果并发执行，不只是“偶尔会错”，而是从语义上就缺乏确定性。当前阶段项目没有引入显式资源锁、依赖图或事务模型，因此串行化是最可靠的默认选择。

### 4. schema 层过滤比执行期拒绝更好

如果模型在 schema 中根本看不到某个危险工具，就不会浪费 token 去尝试调用它，也不会制造额外的错误反馈循环。相比“先暴露、运行时再拒绝”，这是一种更高效、更安全的设计。

### 5. Dispatcher 是独立演进点

工具调度会持续变复杂，例如：

- 未来可能要支持显式工具依赖
- 可能要引入资源级锁
- 可能要做 per-tool concurrency limit

把逻辑放进 `ToolDispatcher`，可以避免 `AgentLoop` 演化成一个混合了控制流、安全和资源调度的巨类。

## 备选方案

### 方案 A：所有工具都并发执行

优点：

- 实现最简单
- 理论吞吐最高

缺点：

- 写操作竞争不可控
- 无法体现不同工具的风险等级
- 对宿主机缺乏保护

结论：

不采用。

### 方案 B：所有工具都串行执行

优点：

- 行为最容易推断
- 不会出现并发写冲突

缺点：

- 牺牲大量只读 I/O 的性能收益
- 把安全问题和性能问题混成一个维度处理，过于保守

结论：

不采用。

### 方案 C：权限分层 + 只读并发 + 写操作串行

优点：

- 在安全和性能之间取得合理平衡
- 默认值更稳
- 能支持后续渐进扩展

缺点：

- 调度逻辑更复杂
- 某些理论上可安全并发的写操作会被保守串行化

结论：

采用。

## 影响

正面影响：

- 工具安全边界在 runtime 层显式化
- 模型不会看到当前 session 不允许使用的工具
- 只读型工具仍可获得并发收益
- 调度复杂度从主循环中拆分出去

负面影响：

- 多个写操作的总耗时会按串行累计
- 当前没有做到更细粒度的资源冲突分析
- 若工具权限标注错误，调度行为也会随之错误

## 结论

`agent-core` 接受“权限分层 + schema 过滤 + 只读并发 + 写操作串行”的工具执行模型。

这项决策的重点不是把工具调用“跑起来”，而是给 Agent runtime 一个更可解释、更安全、更容易扩展的默认执行边界。
