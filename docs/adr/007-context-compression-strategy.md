# ADR-007：Context 压缩策略取舍

## 状态

已接受

## 背景

进入 Day 4 后，`AgentLoop` 已经具备多 step 运行、工具调用和 session 恢复能力。此时如果继续采用“每次都把全部 `messages` 原样喂回模型”的朴素策略，会出现两个明显问题：

1. 输入 token 成本会随着 step 数线性甚至超线性上升  
   一次任务执行到 20~30 步后，单次请求的输入体积会越来越大，导致后续每一轮调用都更贵。

2. 容易触达模型 context window 上限  
   当历史消息不断累积时，模型可能直接返回 `context_length_exceeded`，使运行时在没有压缩机制的情况下无法继续执行。

此外，当前运行时还有一个约束：消息历史中不仅有普通文本，还有 `tool_use` / `tool_result` 这类有引用关系的结构化消息。压缩策略如果只按“最近 N 条消息”截断，可能会产生 dangling reference，例如保留了 `tool_result` 却丢掉了对应的 `tool_use`，从而破坏后续请求的合法性和可理解性。

因此，项目需要一个既能控制 token 成本、又能保持消息结构一致性的上下文压缩策略。

## 决策

我们决定采用混合式 context 压缩策略：

1. 不在每次调用前无条件压缩，而是在估算 token 超过阈值后才触发压缩。
2. 压缩后保留三部分信息：
   - `system_prompt`：始终保留，不参与摘要
   - 最近的原子消息组（tail）：原样保留
   - 中间历史（middle）：交给摘要模型压缩成一条 summary message
3. summary message 使用 `assistant` 角色，而不是 `user` 或 `system`。
4. 对消息切分不按单条 `Message` 处理，而是先构造成不可拆分的原子组：
   - 普通文本消息单独成组
   - 含 `ToolUseContent` 的 `assistant` 消息，和紧随其后的 `tool` 消息视为同一组
5. 中间历史通过独立的摘要调用压缩，默认允许使用比主模型更便宜的小模型。
6. token 估算通过 `TokenEstimator` 协议抽象，默认实现使用轻量估算，后续可切换到更精确的模型专用计数器。

## 理由

### 1. 纯滑动窗口太便宜，但信息丢失不可接受

只保留最近 N 条消息的实现最简单，但它会完全丢失早期的用户目标、关键决策和工具结果。对简单聊天场景也许够用，但对多 step agent 任务来说，早期约束常常决定后续行为，不能直接丢弃。

### 2. 全量摘要虽然节省 token，但会损失局部精度

把全部历史都压缩成一段摘要，能够有效控制体积，但它会同时牺牲“最近对话的细节精度”和“工具调用的结构信息”。对于 agent loop 来说，最后几轮往往正处在任务执行的关键阶段，这部分不应再二次压缩。

### 3. “摘要中间，保留尾部”更符合 agent 的真实信息分布

对长任务而言：

- 头部通常包含系统约束和全局目标
- 中间通常是可概括的执行历史
- 尾部通常是即将影响下一轮推理的高价值局部上下文

因此，采用“保留 tail，摘要 middle”的混合式方案，在信息保留与 token 控制之间更均衡。

### 4. `tool_use` / `tool_result` 必须被视为结构单元

运行时消息不是纯文本序列。`tool_result` 对 `tool_use_id` 有显式引用关系，切分时如果把它们拆开，不仅会降低模型理解能力，还可能在某些 provider 下直接违反请求格式约束。

因此，压缩策略不能只按“最近几条消息”操作，而必须先按原子消息组建模。

### 5. token 估算必须可插拔

Day 4 的目标是先建立稳定边界，而不是一开始就绑定某一家 tokenizer 实现。将 token 估算抽象成 `TokenEstimator` 协议，允许：

- 默认使用便宜、稳定、零依赖的粗估算法
- 后续按 provider 切换到 `tiktoken`、`anthropic.count_tokens` 等精确实现

这样可以先把压缩流程跑通，再逐步提升精度。

## 备选方案

### 方案 A：永远不压缩

优点：

- 实现最简单
- 保留全部原始上下文

缺点：

- token 成本会持续上升
- 迟早撞到 context window 上限
- 长任务无法稳定运行

结论：

不采用。

### 方案 B：纯滑动窗口，只保留最近 N 条消息

优点：

- 实现成本低
- 推理路径清晰

缺点：

- 早期目标和关键决策会被彻底丢弃
- 无法自然处理 `tool_use` / `tool_result` 配对问题
- 对 agent 任务的信息保真度太差

结论：

不采用。

### 方案 C：全部历史摘要成一条消息

优点：

- 最节省 token
- 上下文长度增长最可控

缺点：

- 最近轮次的细节也会丢失
- 工具调用结构会被文本化
- 压缩后很难保留局部精确上下文

结论：

不采用。

### 方案 D：保留尾部原子组 + 压缩中间历史

优点：

- 兼顾 token 控制和信息保真
- 最近关键上下文保留完整
- 能正确处理工具调用配对问题
- 易于接入持久化与恢复机制

缺点：

- 实现复杂度高于纯滑动窗口
- 仍然依赖摘要模型质量
- 需要额外的 token 估算与分组逻辑

结论：

采用。

## 实现约束

当前实现遵守以下约束：

1. `system_prompt` 由 `AgentLoop` 单独传给 provider，不放入 `messages` 列表中；压缩时只对 `messages` 工作集操作，但 token 估算需要把 `system_prompt` 计入。
2. summary message 采用 `assistant` 角色，避免被误当成新的用户指令。
3. 压缩发生时，`ContextManager` 只返回新的 `messages`；是否同步写回 `SessionStore` 由 `AgentLoop` 决定。
4. 当压缩发生时，会发出 `ContextCompressed` 事件，供后续 observability 和 trace 消费。

## 影响

正面影响：

- 控制长任务中的输入 token 成本
- 降低触达 context window 上限的概率
- 为 session 恢复保留更稳定的工作集
- 为后续接入更精确 tokenizer 和更便宜摘要模型预留清晰边界

负面影响：

- 新增一次额外的摘要调用，会带来延迟和成本
- 摘要质量会影响后续推理效果
- 增加了上下文分组、压缩和持久化同步的实现复杂度

## 后续

后续可沿以下方向继续演进：

1. 将粗估 token 计数替换为 provider 级精确计数
2. 引入更明确的压缩触发阈值与目标阈值，减少频繁压缩抖动
3. 结合 trace 数据评估摘要质量与压缩收益
4. 如果未来进入知识密集场景，再评估基于 embedding 或重要性评分的选择性保留策略

## 结论

我们接受“原子消息组 + 中间摘要 + 最近 tail 保留”的混合式 context 压缩策略，作为当前 `agent-core` 长任务运行的默认方案。

这项决策的核心不是追求最强压缩率，而是：

- 在 token 成本可控的前提下维持运行稳定性
- 避免破坏 `tool_use` / `tool_result` 的结构关系
- 为 session 恢复、trace 重放和后续 observability 提供一致的上下文模型
