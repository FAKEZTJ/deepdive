# ADR-001：为什么需要自定义 Provider 抽象

## 状态

已接受

## 背景

本项目的目标不是简单封装某一家模型厂商的 SDK，而是构建一个能够支撑 Agent Runtime 的核心层。这个核心层需要面对以下现实问题：

1. 不同模型厂商的协议差异很大  
   以 OpenAI 和 Anthropic 为例，它们在消息格式、工具调用格式、流式事件模型、错误类型上都有显著差异。

2. 业务层不应该理解厂商细节  
   如果上层代码直接依赖 SDK 的原生消息结构，那么工具执行、上下文管理、流式消费、错误重试等逻辑都会被厂商协议污染。

3. 未来存在多 Provider 并存或切换的需求  
   即使当前只接一个厂商，后续接入第二家时，也常常会暴露出边界没有抽清的问题。越晚抽象，迁移成本越高。

4. Agent Runtime 需要统一的内部语义  
   对运行时来说，更重要的是“用户消息”“工具调用”“工具结果”“文本增量”“流结束”等统一语义，而不是某个 SDK 的字段命名。

因此，项目需要一层明确的 Provider 抽象，把“内部统一语义”和“外部厂商协议”隔离开。

## 决策

我们决定引入自定义 Provider 抽象层，并采用 Adapter 模式接入各家模型厂商。

具体决策如下：

1. 定义统一的内部数据模型  
   使用 `Message`、`TextContent`、`ToolUseContent`、`ToolResultContent`、`CompletionResponse`、`StreamEvent` 等类型作为项目内部的标准语义。

2. 定义统一的 Provider 接口  
   使用 `LLMProvider` 作为抽象基类，对外只暴露：

   - `chat(...)`
   - `chat_stream(...)`

3. 每个厂商通过独立 Provider 适配  
   当前包括：

   - `OpenAIProvider`
   - `AnthropicProvider`

   这些 Provider 负责：

   - 请求格式转换
   - 响应格式转换
   - 工具调用协议转换
   - 流式事件归一化
   - 厂商异常映射

4. 运行时与业务层只依赖内部抽象  
   Runtime、Tool Executor、Context、CLI 等上层模块不直接依赖厂商 SDK。

## 取舍理由

我们选择自定义抽象，而不是直接把某个 SDK 当作项目的核心接口，主要基于以下考虑：

### 1. 降低厂商耦合

如果直接在业务代码中使用 OpenAI 或 Anthropic 的原生类型，那么后续更换 Provider 时，不只是网络调用层要变，消息解析、工具调用、流式处理和错误处理都要跟着一起变。

自定义抽象可以把变化限制在 Provider 层。

### 2. 保持运行时语义稳定

Agent Runtime 关注的是统一语义，而不是厂商字段。比如：

- 模型是否在请求工具
- 工具参数是否在流式增长
- 一段文本是否已经结束
- 最终停止原因是什么

这些语义应该由项目内部定义，而不是由厂商 API 决定。

### 3. 提升测试可控性

统一抽象后，测试可以围绕内部契约编写，而不是依赖 SDK 细节。这样可以更容易验证：

- Provider 的格式转换是否正确
- 流式事件顺序是否稳定
- 异常映射是否一致

### 4. 为后续扩展预留空间

后续如果增加：

- 新 Provider
- 本地模型
- Mock Provider
- Router / Fallback / Retry Provider

都可以在现有抽象之上扩展，而不必重写上层逻辑。

## 备选方案

### 方案 A：直接使用 OpenAI SDK 作为核心接口

优点：

- 初期开发最快
- 少一层抽象

缺点：

- 业务层直接绑定 OpenAI 协议
- 后续接入 Anthropic 或其他 Provider 成本高
- 流式事件与工具调用逻辑难以复用

结论：

不采用。该方案只适合单 Provider、短生命周期、无统一 runtime 诉求的项目。

### 方案 B：仅做轻量 helper，不做正式抽象层

优点：

- 上手快
- 看起来更“轻”

缺点：

- 边界不清晰
- 业务层仍会逐步侵入 SDK 细节
- 很容易演化成“到处都是条件分支”的半抽象状态

结论：

不采用。它无法形成稳定的架构边界。

### 方案 C：定义统一 Provider 抽象并适配各家协议

优点：

- 内部语义统一
- 多 Provider 可扩展
- 测试与运行时边界清晰

缺点：

- 初期实现成本更高
- 需要维护一层适配逻辑
- 某些厂商特性无法原样暴露，需要经过抽象取舍

结论：

采用。这是对当前项目目标最匹配的方案。

## 影响

### 正面影响

- 上层模块不依赖具体厂商 SDK
- Provider 切换或新增 Provider 的成本降低
- 流式事件和工具调用协议可以统一消费
- 错误处理、重试、监控等横切逻辑更容易集中管理

### 负面影响

- 需要维护抽象层与适配层代码
- 某些厂商特有能力可能无法第一时间暴露到统一接口
- 抽象设计如果过早或过度，可能带来额外复杂度

## 当前落地结果

截至 ADR 编写时，项目已经初步落地该决策：

- 已定义统一类型模型
- 已定义 `LLMProvider`
- 已实现 `OpenAIProvider`
- 已实现 `AnthropicProvider`
- 已对工具调用与流式事件进行了统一转换

这说明该决策不是纯理论预留，而是已经成为当前代码结构的核心前提。

## 后续约束

后续新增 Provider 或 Runtime 能力时，应遵守以下约束：

1. 不在业务层直接传播厂商 SDK 类型
2. 新 Provider 必须对齐统一 `chat` / `chat_stream` 契约
3. 新流式行为必须经过测试验证
4. 如果某个厂商特性无法自然落入现有抽象，应先讨论是否扩展内部模型，而不是直接绕过抽象

## 结论

我们接受“自定义 Provider 抽象”作为 `agent-core` 的基础架构决策。

这项决策的核心价值不在于“屏蔽 SDK”本身，而在于：

- 为运行时提供稳定的内部语义
- 为多 Provider 扩展提供清晰边界
- 将协议差异限制在 Provider Adapter 层

这是当前项目继续演进为 Agent Runtime Core 的必要前提。
