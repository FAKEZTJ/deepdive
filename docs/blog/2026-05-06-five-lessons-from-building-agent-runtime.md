# 实现 Agent Runtime 时我踩过的 5 个坑

很多人第一次做 Agent，都会先从“调通模型 + 接个工具”开始。我也一样。但一旦目标从“跑通一次 demo”变成“把一个 agent 连续跑十几步、能恢复、能调试、能换模型”，坑会很快从 prompt 层转移到 runtime 层。

这篇文章不讲“如何做一个聊天机器人”，只讲我在做 `agent-core` 这类 Agent Runtime 时踩过的 5 个典型坑。

## 1. 把厂商 SDK 当成系统核心接口

最早期的实现很容易长这样：

- 业务层直接拼 OpenAI 的 message 格式
- 工具调用直接读取 `tool_calls`
- 流式输出直接消费某家 SDK 的 delta

短期看起来没问题，因为你只接了一家模型，而且 demo 也能跑。但一旦你要：

- 接 Anthropic 或 DeepSeek
- 把工具调用抽象成统一闭环
- 给流式文本和工具参数做统一事件协议

你就会发现，所谓“业务逻辑”里混进去的其实都是供应商细节。

我后面做的第一个大修正，就是先定义内部统一语义，再让 Provider 做适配。也就是把系统真正关心的东西收敛成这些概念：

- `Message`
- `ToolUseContent`
- `ToolResultContent`
- `CompletionResponse`
- `StreamEvent`

这样上层 runtime 处理的是“模型请求工具了”“文本块结束了”“这一步 stop 了”，而不是“某个 SDK 的某个字段是不是这次又改名了”。

经验总结：

- SDK 是集成边界，不应该是系统边界。
- 越早把内部语义和外部协议拆开，后面越不容易返工。

## 2. 把工具调用当成一段 JSON，而不是状态机的一部分

很多 Agent demo 都默认模型返回工具参数后，宿主直接执行一下，再把结果塞回去。这个流程没错，但它经常把两个关键问题忽略掉了：

1. 工具有没有权限边界？
2. 多个工具一起出现时，谁能并发，谁必须串行？

如果没有权限模型，工具就只剩“能不能被模型调用”这一层语义。那 runtime 最终会变成一个很危险的东西：模型只要写出名字，就能拿到任意宿主能力。

如果没有调度模型，也会很快出事。只读工具并发通常没问题，但写文件、改状态、执行 shell 这些操作如果默认并发，行为会非常不稳定。

所以我后来明确做了三件事：

- 给工具定义 `read_only / write / dangerous` 权限档位
- session 启动时用白名单限制能暴露给模型的能力
- 调度时只并发 `read_only`，把 `write / dangerous` 串行化

这看起来有点“重”，但好处是 runtime 的安全边界开始可解释了。

经验总结：

- 工具调用不是普通输出格式，而是运行时状态转换。
- 如果不在 runtime 层显式建模权限和顺序，复杂度最后只会散落到业务层。

## 3. 只想着“上下文够不够”，没想着“结构会不会坏”

做长任务时，几乎一定会碰到上下文压缩。最直觉的做法是：

- 只保留最近 N 条消息
- 或者把全部历史摘要成一段文本

问题在于，Agent 的历史不是普通聊天记录。里面会有：

- `assistant` 发出的 `tool_use`
- `tool` 返回的 `tool_result`
- 二者之间的引用关系

如果你只是按“最近几条消息”截断，极有可能保留了 `tool_result`，却把对应的 `tool_use` 删掉了。对人类读者来说，这只是有点迷惑；对 runtime 来说，这可能是结构直接损坏。

后来我改成了“原子消息组”的思路：

- 普通文本消息单独成组
- `assistant(tool_use)` + 紧随其后的 `tool` 消息视为一组

然后在压缩时：

- 保留最近几组原始上下文
- 把更早的 middle history 摘要成一条 summary message

这比简单滑动窗口复杂，但它至少没有破坏 runtime 的结构前提。

经验总结：

- 长上下文问题不只是 token 问题，还是结构问题。
- 对 Agent 来说，压缩策略必须理解工具调用关系。

## 4. 以为“把消息落盘”就等于“支持恢复”

这是我踩得最深的一个坑。

一开始很容易觉得：既然 `messages` 是模型输入，那我把每条消息实时写进数据库，不就能恢复了吗？

后来发现这会制造大量“半步状态”：

- assistant 已经请求了工具，但工具结果还没回来
- tool result 已经写了，但这一轮 step 其实还没完成
- 压缩刚发生过，数据库里的消息工作集和内存状态并不一致

这些状态在不中断时没问题，但一旦进程退出、崩溃或者你真的调用 `resume()`，它们都会变成脏上下文。

所以后面我把“运行事件”和“可恢复状态”分开了：

- `events` 表实时记录过程
- `messages` 只在 step 完成后 checkpoint
- `sessions` 记录总步数、usage、cost 和状态

这样恢复语义就变得明确了：只能恢复到最近一个稳定 step 边界，而不是任意中间点。

听上去能力变弱了，但它换来的是语义可靠。

经验总结：

- 日志能回放，不代表状态能恢复。
- 恢复语义必须围绕“稳定边界”设计，而不是围绕“我已经把什么写进数据库了”设计。

## 5. 把 trace、日志和业务事件混成一团

一开始做观测时，最容易出现两种极端：

- 只有打印日志，调试全靠肉眼翻文本
- 只接 OpenTelemetry，把所有事情都寄托给外部 backend

前者的问题是结构化不够，后者的问题是本地体验会很差。尤其是做 runtime，本地排查单次 session 是高频动作，如果看一次 trace 还要先起一套基础设施，门槛太高了。

我最后采取的是双通道思路：

- SQLite `events` 作为持久化事实日志
- OTel span 作为运行时层次视图

换句话说：

- “发生了什么”看 SQLite
- “这些事情是如何嵌套的、各自耗时多少”看 span

然后在事件里把 `trace_id / span_id / parent_span_id` 一起存下来，让两套视图能对齐。

这样做的好处是：

- 本地可以直接 `trace session_id`
- 生产环境仍能接标准 OTel
- 运行日志、成本统计和业务事件也能关联起来

经验总结：

- 业务事件和 tracing 是互补关系，不是二选一。
- 如果你做的是本地优先 runtime，离线可读的事实日志非常重要。

## 最后

回头看，这 5 个坑有一个共同点：它们都不是“模型不够强”导致的，而是运行时边界没想清楚导致的。

真正让 Agent 系统变稳定的，往往不是更复杂的 prompt，而是这些更偏工程的问题：

- 内部语义是不是独立于供应商协议
- 工具调用有没有权限和调度边界
- 上下文压缩会不会破坏结构
- 恢复语义是不是围绕稳定状态设计
- 可观测性是不是本地就能闭环

如果你的 Agent 还停留在 demo 阶段，这些问题看起来都像“以后再说”。但如果你已经开始做长任务、多步工具调用、可恢复和可调试的 runtime，它们迟早会变成必须回答的问题。

对我来说，做 `agent-core` 最大的收获，不是“又接了一个模型”，而是越来越清楚：Agent Runtime 的核心难点，根本不在调用 LLM，而在管理围绕 LLM 发生的一切。
